## Step 1: コンテストの仕組みを理解する

### 匿名化と再識別の攻防戦

PWScup は、**データのプライバシー保護**をテーマにしたコンテストです。

身近な例えで考えてみましょう。あなたが病院の患者データを統計分析のために公開したいとします。そのまま公開すると「30歳、男性、東京都新宿区在住」という情報から個人が特定されてしまうかもしれません。そこで、年齢を「30代」に丸めたり、住所を「東京都」にぼかしたりします。これが**匿名化**です。

一方、攻撃者の立場で考えると、選挙人名簿のような公開情報（補助知識）を使って、匿名化されたデータと個人を結びつけようとするかもしれません。これが**再識別**です。

このコンテストでは、この攻防を2つの部門で競います:

| 部門 | 役割 | 目標 |
|------|------|------|
| **匿名化部門（守り）** | データを匿名化する | 有用性を保ちつつ、安全なデータを作る |
| **再識別部門（攻め）** | 匿名化データを復元する | 匿名化されたデータから元の個人を特定する |
| **総合部門** | 両方を行う | 匿名化と再識別の両方で高得点を狙う |

### スコアの仕組み

#### 匿名化スコア

匿名化のスコアは **有用性 (U)** と **安全性 (S)** の積で決まります。

```
匿名化スコア = U × S
```

**これが最大のポイントです。** 片方が 0 なら全体が 0 になります。つまり、「完全にランダムなデータを出して安全性だけ高くする」「元データそのまま出して有用性だけ高くする」といった極端な戦略は通用しません。

##### 有用性スコア U（匿名化データがどれだけ使えるか）

| 指標 | 重み | 説明 |
|------|------|------|
| 分布距離 | 0.3 | 元データと匿名化データの分布がどれだけ似ているか |
| 相関保存 | 0.3 | カラム間の相関関係がどれだけ保たれているか |
| クエリ精度 | 0.2 | 「30代男性の平均年収は?」のような集計クエリの結果が一致するか |
| ML有用性 | 0.2 | 匿名化データで学習した機械学習モデルの性能がどれだけ保たれるか |

##### 安全性スコア S_auto（データがどれだけ安全か）

3つのプライバシー指標の平均です:

| 指標 | 説明 |
|------|------|
| k-匿名性 | 同じ準識別子の組み合わせを持つレコードが少なくとも k 件あるか |
| l-多様性 | 同じグループ内で機微属性の値が少なくとも l 種類あるか |
| t-近接性 | 各グループの機微属性の分布が全体分布とどれだけ近いか |

```
S_auto = (k-匿名性スコア + l-多様性スコア + t-近接性スコア) / 3
```

#### 再識別スコア

再識別では、匿名化データの各レコードが元データのどのレコードに対応するかを予測します。

| 指標 | 説明 |
|------|------|
| Precision | 予測した対応のうち、正解だった割合 |
| Recall | 正解の対応のうち、予測できた割合 |
| F1 | Precision と Recall の調和平均 |

匿名化の難易度が高いほど、再識別スコアに加重がかかります（より困難な匿名化を破った方が高評価）。

### データの種類

コンテストで扱うデータには、カラムごとに役割があります:

| 役割 | 説明 | 例 |
|------|------|-----|
| **識別子 (ID)** | 個人を一意に特定する。匿名化時に必ず削除 | id |
| **準識別子 (QI)** | 単独では個人特定できないが、組み合わせると特定の手がかりになる | age, gender, zipcode, occupation, education |
| **機微属性 (SA)** | 知られたくない情報。保護対象 | disease, salary |
| **非機微属性** | 特に保護の必要がないが、有用性評価に使われる | hobby |

## Step 3: ベースライン匿名化を試す

### ベースラインアルゴリズムの仕組み

`examples/anonymize_example/algorithm.py` にベースラインの匿名化アルゴリズムが実装されています。このアルゴリズムは以下のことを行います:

1. **識別子 (id) の削除**: 匿名化の大前提
2. **数値QI（age）の汎化**: 値を5段階に丸める（例: 52歳 → 49歳、42歳 → 49歳）
3. **カテゴリQI の汎化**:
   - 郵便番号: 上1桁のみ残す（例: `163-5060` → `100-0000`）-- 地方レベルまで粗くする
   - 職業・学歴: 出現頻度の上位3値以外を最頻値にマージ
4. **数値SA（salary）へのノイズ付加**: ラプラスノイズを加えて値をぼかす
5. **サプレッション**: k-匿名性 k < 2 のレコード（同じQIの組が1件しかないもの）を削除

### 実行してみる

以下のコマンドで匿名化評価を実行します:

```bash
uv run pwscup evaluate anonymize examples/anonymize_example/ \
  --data-dir data/sample \
  --schema-path data/schema/schema.json
```

- **相関保存 0.995**: カラム間の相関がほぼ完全に保たれている
- **クエリ精度 0.944**: 集計クエリの精度が非常に高い

有用性は高いです。匿名化があまり強くないため、データの性質がよく残っています。

#### 安全性スコア: 0.151（かなり低い）

- **k=2（スコア: 0.20）**: 同じQIの組み合わせが最低2件。k=2は最低限のレベル

有用性は高いものの、安全性が低いためスコアが伸びていません。**これがまさに U × S 設計の狙い**です。安全性を犠牲にして有用性だけ高めても、最終スコアは低くなります。

## Step 4: ベースライン再識別を試す

### ベースラインアルゴリズムの仕組み

`examples/reidentify_example/algorithm.py` には、準識別子の**最近傍マッチング**による再識別アルゴリズムが実装されています。

仕組みは単純です:
1. 補助知識（外部から入手できる情報: 氏名、年齢、性別、郵便番号、職業、学歴）を持っている
2. 匿名化データの各レコードと補助知識のレコードを、QI（準識別子）の値で比較する
3. 最も「距離」が近い組み合わせを「同一人物」と判定する

距離の計算方法:
- **数値カラム**: 値の差をカラムのレンジで正規化（例: 年齢差10歳 / レンジ72 = 0.139）
- **カテゴリカラム**: 完全一致なら0、不一致なら1

### 実行してみる

まず匿名化データを生成し、その後再識別評価を行います:

```bash
# Step 1: 匿名化データを生成
uv run python examples/anonymize_example/algorithm.py \
  data/sample/sample_original.csv \
  data/schema/schema.json \
  /tmp/anonymized.csv

# Step 2: 再識別評価を実行
uv run pwscup evaluate reidentify examples/reidentify_example/ \
  --anon-csv /tmp/anonymized.csv \
  --auxiliary-csv data/auxiliary/sample_auxiliary.csv \
  --ground-truth-path data/auxiliary/sample_ground_truth.json \
  --schema-path data/schema/schema.json
```

実行結果:

## Step 5: スコアを改善してみよう

### 匿名化の改善ヒント

Step 3 の結果を振り返ると、有用性は高い（0.904）が安全性が低い（0.151）ことが課題でした。安全性を高める方向で改善してみましょう。

#### ヒント 1: 汎化の粒度を調整する

現在のベースラインでは年齢を5段階に丸めていますが、これを10段階にするなど粒度を粗くすれば、同じQIの組み合わせを持つレコードが増え、k-匿名性が向上します。

```python
# 例: 5段階 → 10段階に変更
granularity = max((col_range[1] - col_range[0]) // 10, 1)
```

ただし、粗くしすぎると有用性が下がります。**ちょうどいいバランス**を見つけることが鍵です。

#### ヒント 2: ノイズの大きさを調整する

salary へのラプラスノイズの大きさ（scale パラメータ）を変えることで、安全性と有用性のバランスを調整できます。

```python
# 現在: レンジの5%
scale = (col_range[1] - col_range[0]) * 0.05
# 大きくすれば安全だが、有用性が下がる
scale = (col_range[1] - col_range[0]) * 0.10
```

#### ヒント 3: サプレッション vs 汎化のバランス

- **サプレッション**（レコード削除）: 外れ値を削除して安全性を上げるが、レコード数が減る
- **汎化**（値の丸め）: レコードは残るが、細かい情報が失われる

現在は k < 2 で削除していますが、k の閾値を上げたり、汎化をより強くして削除を減らしたりする戦略が考えられます。

#### ヒント 4: カテゴリ汎化の工夫

郵便番号を上1桁だけ残すのは粗すぎるかもしれません。上3桁にするなど、もう少し細かい汎化を試してみましょう:

```python
# 上3桁を残す（市区町村レベル）
df[name] = df[name].astype(str).str[:3] + "-0000"
```

### 再識別の改善ヒント

Step 4 の結果ではほぼ当てられていませんでした（F1 = 0.010）。大幅な改善が期待できます。

#### ヒント 1: 機微属性も活用する

ベースラインはQI（準識別子）のみで距離を計算していますが、匿名化データに残っている機微属性（disease, salary）や非機微属性（hobby）も手がかりとして使えます。

#### ヒント 2: 確信度に基づくフィルタリング

距離が近い候補が複数ある場合、確信度が低い予測を除外することで Precision を改善できます。

```python
# 確信度が閾値以上の場合のみ予測する
if confidence > 0.5:
    mappings.append(...)
```

Precision を上げると Recall が下がるトレードオフがありますが、F1（調和平均）で最適なバランスを探しましょう。

#### ヒント 3: より高度なマッチングアルゴリズム

- **ハンガリアン法**: 全体最適な1対1マッチングを求める
- **確率的マッチング**: 各属性の一致確率を考慮した重み付きマッチング
- **グループ情報の活用**: 匿名化の汎化パターンを推定してマッチング精度を上げる

### トレードオフを理解する

匿名化において最も重要な概念は**トレードオフ**です:

```
          安全性 高
              |
              |    理想は右上
              |        *
              |
              +------------ 有用性 高
```

- **右下（有用性だけ高い）**: 元データそのまま。スコア = 高い有用性 × 0 = **0**
- **左上（安全性だけ高い）**: ランダムデータ。スコア = 0 × 高い安全性 = **0**
- **右上（両方高い）**: これを目指す。巧みな匿名化技術が必要

再識別側にも同様のトレードオフがあります:

- **たくさん予測する**: Recall は上がるが Precision が下がる
- **厳選して予測する**: Precision は上がるが Recall が下がる
- **F1 を最大化**: 両者のバランスが取れた戦略

## Step 6: 提出の流れ（概要）

### 提出物のディレクトリ構成

提出ディレクトリには以下のファイルが必要です:

```
my_submission/
  algorithm.py      # メインの実行ファイル（必須）
  metadata.json     # チーム名・部門・説明（必須）
  requirements.txt  # 追加の依存パッケージ（任意）
```

`metadata.json` の例:

```json
{
  "team": "my_team",
  "division": "anonymize",
  "description": "k-匿名性を重視した汎化ベースの匿名化"
}
```

### 提出コマンド

```bash
# 匿名化アルゴリズムの提出
uv run pwscup submit anonymize my_submission/ \
  --team-name my_team \
  --no-use-docker

# 再識別アルゴリズムの提出
uv run pwscup submit reidentify my_submission/ \
  --team-name my_team \
  --no-use-docker
```

> **ヒント**: 提出前に必ず `evaluate` コマンドでローカルテストを行いましょう。

### リーダーボードの確認

提出後のランキングは以下のコマンドで確認できます:

```bash
# 匿名化部門のリーダーボード
uv run pwscup leaderboard --division anonymize

# 再識別部門のリーダーボード
uv run pwscup leaderboard --division reidentify

# 上位10件に絞る
uv run pwscup leaderboard --division anonymize --top 10
```

## 付録: 用語集

| 用語 | 説明 |
|------|------|
| **QI（準識別子）** | Quasi-Identifier。単独では個人を特定できないが、組み合わせると特定の手がかりになるカラム（例: 年齢、性別、郵便番号） |
| **SA（機微属性）** | Sensitive Attribute。他人に知られたくない情報（例: 疾患名、年収）。匿名化の主要な保護対象 |
| **k-匿名性** | データセット内で、同じQIの値の組み合わせを持つレコードが少なくとも k 件存在すること。k が大きいほど安全 |
| **l-多様性** | 同じQIグループ内で、機微属性の値が少なくとも l 種類あること。l=1 だと全員同じ病名で特定される |
| **t-近接性** | 各QIグループ内の機微属性の分布が、データセット全体の分布と t 以下の距離であること。t が小さいほど安全 |
| **EMD（Earth Mover's Distance）** | 2つの確率分布の間の距離を測る指標。t-近接性の計算に使われる。「土を動かすコスト」の比喩で、一方の分布をもう一方に変形するのに必要な最小コストを表す |
| **汎化** | データの値をより一般的な値に置き換える手法（例: 年齢 52 → 50代、郵便番号 163-5060 → 100-0000） |
| **サプレッション** | k-匿名性を満たさないなど、安全でないレコードを削除する手法 |
| **ラプラスノイズ** | 差分プライバシーなどで使われるノイズ。ラプラス分布に従うランダムな値を加えてデータを撹乱する |
| **Precision（適合率）** | 予測した対応のうち、実際に正解だった割合。「予測の正確さ」 |
| **Recall（再現率）** | 正解の対応のうち、正しく予測できた割合。「見つけ漏れの少なさ」 |
| **F1スコア** | Precision と Recall の調和平均。両方のバランスを表す指標 |
| **補助知識** | 攻撃者が外部から入手可能な情報（例: 選挙人名簿、SNSプロフィール）。再識別に使用する |
