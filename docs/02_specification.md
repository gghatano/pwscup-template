# PWSCUP データ匿名化・再識別コンテスト 仕様書

## 1. コンテスト概要

### 1.1 目的

データ匿名化技術と再識別技術の攻防を通じて、プライバシー保護技術の発展を促進するコンテスト環境を構築する。

### 1.2 基本構造

本コンテストは**匿名化部門**と**再識別部門**の2部門で構成される「攻防戦」形式のコンテストである。

- **匿名化部門（守り）**: 生データを匿名化し、有用性を保ちつつ安全なデータを生成する
- **再識別部門（攻め）**: 匿名化されたデータに対して攻撃し、元データとの対応を復元する
- **総合部門**: 両部門の順位を統合した総合力で順位を決定する

匿名化部門の出力が再識別部門の入力になる**相互評価構造**が最大の特徴である。

### 1.3 参加形態

- 個人またはチーム（1〜3名）で参加可能
- 匿名化のみ・再識別のみ・両方の3形態で参加可能
- 両部門参加を推奨するが、片方のみでも部門別順位に参加できる

## 2. データセット仕様

### 2.1 データ方針

**合成データを使用する**。理由：

- 個人情報保護法第43条の再識別禁止規定への抵触リスクを回避
- GDPR対応の負荷を排除
- 正解データ（グランドトゥルース）を自由に設計でき、評価が厳密
- 倫理審査の負荷を最小化

合成データは実データの統計的特性（分布、相関構造、欠損パターン）を再現するよう設計する。「本データは完全に合成されたものであり、実在の個人とは無関係です」と明示する。

### 2.2 データスキーマ

| 区分 | 属性例 | 数 | 役割 |
|------|--------|-----|------|
| 識別子 | id | 1 | 匿名化前に運営が削除 |
| 準識別子 (QI) | 年齢, 性別, 郵便番号, 職業, 学歴 | 5〜8 | 外部知識と結合して個人を特定しうる属性 |
| 機微属性 (SA) | 疾病名, 年収 | 1〜2 | 保護対象となる属性 |
| 非機微属性 | 趣味, 購買カテゴリ | 2〜3 | 有用性評価に利用 |

### 2.3 データ規模

| パラメータ | 値 | 理由 |
|------------|-----|------|
| レコード数 | 10,000〜50,000 | 匿名化が容易すぎず、計算コストが参加障壁にならない範囲 |
| 準識別子の数 | 5〜8 | 等価クラスの細分化により適度な難易度を確保 |
| 機微属性の数 | 1〜2 | 評価の明確さのため |
| カーディナリティ | 混在 | 低（性別: 2値）と高（年齢: 連続値）を混ぜる |

### 2.4 データフォーマット

- **形式**: CSV（UTF-8）
- **スキーマ定義**: `schema.json` で各カラムの型・属性分類・値域・汎化階層を定義

```json
{
  "columns": [
    {
      "name": "age",
      "type": "numeric",
      "role": "quasi_identifier",
      "range": [0, 120],
      "hierarchy": "hierarchies/age.csv"
    },
    {
      "name": "disease",
      "type": "categorical",
      "role": "sensitive_attribute",
      "domain": ["diabetes", "flu", "cancer", "..."]
    }
  ],
  "quasi_identifiers": ["age", "gender", "zipcode", "occupation", "education"],
  "sensitive_attributes": ["disease", "salary"]
}
```

### 2.5 データセットの種類

| 用途 | 公開範囲 | 説明 |
|------|----------|------|
| サンプルデータ | 全参加者に公開 | 練習用。本番と同一スキーマ・類似分布 |
| 予選データ | 予選参加者に公開 | 予選期間中に配布 |
| 本選データ | 本選参加者にのみ配布 | 本選開始時に配布。非公開 |

### 3.1 提出インターフェース

参加者はPython関数を実装し、規定のインターフェースに従って提出する。

#### 匿名化部門

```python
def anonymize(input_csv_path: str, schema_path: str, output_csv_path: str) -> None:
    """
    入力: 生データCSVのパス, スキーマJSONのパス

    制約: 出力のカラム名・データ型は入力と一致すること（行数は変動可）
    """
```

#### 再識別部門

```python
def reidentify(anon_csv_path: str, auxiliary_csv_path: str, schema_path: str, output_json_path: str) -> None:
    """
    入力: 匿名化済みCSVのパス, 補助知識CSVのパス, スキーマJSONのパス
    出力: マッピングJSONを output_json_path に書き出す
    マッピング形式: [{"anon_row": 0, "original_id": 42, "confidence": 0.95}, ...]

```


## 4. 評価・スコアリング

### 4.1 匿名化部門の評価

匿名化部門のスコアは**有用性**と**安全性**の積で算出する。

```
Score_anon = U × S
```

どちらかが0なら全体も0となり、片方を犠牲にする戦略を構造的に排除する。

#### 4.1.1 有用性スコア (U: 0〜1)

| 指標 | 重み | 説明 |
|------|------|------|
| カラム別分布距離 | 0.3 | 各カラムのEarth Mover's Distance の平均 |
| 相関構造保存度 | 0.3 | 相関行列間のFrobeniusノルムの逆数（正規化） |
| クエリ応答精度 | 0.2 | 事前定義した集計クエリ群（COUNT/SUM/AVG）に対する相対誤差 |
| ML有用性 | 0.2 | 匿名化データで学習したモデルの元テストデータ上での精度維持率 |

各指標を0〜1に正規化し、重み付き平均でUを算出する。重みはコンテスト開始時に公開する。

#### 4.1.2 安全性スコア (S: 0〜1)

二段構成とする。

```
S = 0.4 × S_auto + 0.6 × S_reid
```

**S_auto（静的安全性評価）**:
- k-匿名性の達成度
- l-多様性の達成度
- t-近接性の達成度
- 提出直後に即時算出される暫定スコア

**S_reid（再識別耐性）**:
- 全再識別攻撃者の中で最も成功率が高かった攻撃に対する耐性
- `S_reid = 1 - max(再識別成功率)`
- 再識別ラウンド終了後に確定

**最低基準**: k ≧ 2 を参加条件とし、元データそのままの提出を排除する。

### 4.2 再識別部門の評価

```
Score_reid = Σ_i (precision_i × recall_i × difficulty_i) / Σ_i difficulty_i
```

- precision: 対応付けを主張したレコードのうち正解の割合
- recall: 全レコードのうち正しく対応付けできた割合
- difficulty_i = 1 / (1 - S_auto_i + ε): 安全性の高い匿名化データを破った場合により高い重み

これにより、弱い匿名化ばかり攻撃しても高スコアにならない設計。

### 4.3 総合スコア

```
Rank_total = 0.5 × Rank_anon + 0.5 × Rank_reid
```

- 順位ベースでマージ（スコアのスケールが異なるため）
- 片方のみ参加の場合、不参加部門は「参加者数 + 1」位として扱う
- 同率の場合は提出時刻が早い方を上位
- 両部門参加へのインセンティブを確保

## 5. コンテストフェーズ設計

### 5.1 フェーズ構成

| フェーズ | 期間 | 内容 |
|----------|------|------|
| Phase 0: 登録・チュートリアル | 2週間 | 参加登録、環境構築、ルール理解、チュートリアル学習 |
| Phase 1: 練習ラウンド | 2週間 | サンプルデータで動作確認、ベースライン提出体験 |
| Phase 2: 予選（匿名化提出） | 3週間 | 予選データに対する匿名化アルゴリズムの提出 |
| Phase 3: 予選（再識別攻撃） | 2週間 | Phase 2の匿名化データに対する再識別攻撃 |
| Phase 4: 本選 | 3週間 | 非公開データでの匿名化→再識別の最終評価 |
| Phase 5: 最終審査・表彰 | 1週間 | コードレビュー、再現性確認、結果発表 |

### 5.2 ラウンド制の運用

匿名化→再識別のフェーズを分離することで：
- 匿名化の再提出ループ（攻撃結果を見て再提出）を防止
- 評価の公平性を担保
- 計算量の爆発（M匿名化 × N再識別）を管理可能にする

### 5.3 提出制限

| 項目 | 制限値 |
|------|--------|
| 1日あたりの提出回数 | 各部門5回 |
| フェーズ全体の提出回数 | 各部門50回（予選）、20回（本選） |
| 最終選択提出数 | 各部門2件（最終評価に使う提出を自己選択） |

## 6. リーダーボード

### 6.1 二重リーダーボード方式

| 種別 | 可視性 | 計算対象 |
|------|--------|----------|
| Public LB | 参加者全員に公開 | 有用性 + S_auto のみ（暫定） |
| Private LB | 最終発表時に公開 | 有用性 + S_auto + S_reid（完全版） |

匿名化部門の安全性は再識別結果に依存するため、Public LBではS_autoのみで暫定表示する。

### 6.2 表示項目

**匿名化部門**: 順位, チーム名, 有用性スコア(U), 安全性スコア(S_auto/暫定), 総合スコア, 提出回数

**再識別部門**: 順位, チーム名, 平均再識別スコア, 攻撃成功データ数/全データ数, 提出回数

**総合**: 総合順位, チーム名, 匿名化順位, 再識別順位, 総合ランクスコア

## 7. システムアーキテクチャ

### 7.1 技術スタック

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| CLI | Python + Click/Typer | 参加者との親和性、開発速度 |
| API | FastAPI | 非同期対応、型安全、自動ドキュメント |
| ジョブキュー | Redis + RQ | 軽量。小規模ならsubprocess直接実行も可 |
| DB | SQLite（初期）→ PostgreSQL（拡張時） | ゼロ設定、単一ファイルでバックアップ容易 |
| ORM | SQLAlchemy / SQLModel | DB差し替えが容易 |
| サンドボックス | Docker (docker-py) | 標準的なコンテナ隔離 |
| 評価ロジック | Python (pandas, scipy, scikit-learn) | 統計指標の計算 |
| テスト | pytest + GitHub Actions | 評価ロジックのテストが特に重要 |

### 7.2 実行モード

- **ローカルモード**: サーバーなしでローカルDockerだけで完結。参加者が手元でテスト実行可能
- **サーバーモード**: FastAPI + ジョブキューで非同期評価。大規模運用時に使用

初期実装はローカルモードを優先する。

### 7.3 サンドボックス実行（多層防御）

| 層 | 対策 |
|----|------|
| 第1層 | Dockerコンテナ分離（使い捨て） |
| 第2層 | ネットワーク完全遮断 (`--network=none`) |
| 第3層 | リソース制限（CPU, メモリ, ディスク, プロセス数） |
| 第4層 | 権限制限（非root実行, seccomp/AppArmorプロファイル） |
| 第5層 | データアクセス制御（入力データread-onlyマウント） |

## 8. 参加者体験（UX）

### 8.1 チュートリアル・ベースライン

- 4段階のチュートリアル: データ確認 → 簡単な匿名化体験 → 再識別攻撃体験 → 評価指標理解
- ベースライン実装の配布: 匿名化側はランダムノイズ付加、再識別側はナイーブマッチング
- テンプレートリポジトリを `git clone` → `make run` でスコアが出る体験を提供

### 8.2 ローカル評価ツール

- 提出前に手元でスコアを概算できるスクリプトを提供
- 本番の評価用データとは異なるが、指標の計算ロジックは同一

### 8.3 フィードバック

**提出直後**:
- フォーマットエラー・タイムアウト等の検出結果
- 実行時間とメモリ使用量

**評価完了後（匿名化部門）**:
- 有用性スコア（全体 + 指標別内訳）
- 安全性スコア（S_auto）
- 有用性 vs 安全性のトレードオフ上での自分の位置

**評価完了後（再識別部門）**:
- 再識別成功率（統計のみ。具体的なマッチング結果は非開示）
- 対戦した匿名化手法の難易度ランク

**大会終了後**:
- 上位手法の概要レポート
- 全提出履歴とスコア推移

### 8.4 表彰制度

- **総合優勝**: 両部門の総合力
- **匿名化最優秀賞**: 匿名化部門1位
- **再識別最優秀賞**: 再識別部門1位
- **特別賞**: 最も独創的な手法、新人賞（初参加者限定）等

## 9. セキュリティ・法務・倫理

### 9.1 データの合法性

- 合成データのみ使用し、個人情報保護法・GDPRの規制対象外とする
- コンテストの目的を「匿名化手法の安全性評価」と明確に位置付け、再識別自体が目的ではないことを文書化

### 9.2 利用規約の主要条項

- コンテストで得た再識別技術・知見を実データに対して使用することの**明示的禁止**
- 提供データの外部再配布の禁止
- サンドボックス環境への攻撃・逸脱行為の禁止
- 提出コードの知的財産権は参加者に帰属（コンテスト目的での利用許諾を取得）
- 運営によるコード検査・監査への同意

### 9.3 不正防止

- 同一チームの両部門参加時、自チームの匿名化データに対する再識別スコアは除外
- 提出コードの類似度分析（AST比較）による複数アカウント検出
- 本選上位チームのコード提出義務・再現性検証（許容誤差5%以内）

### 9.4 データ漏洩防止

- 生データはサーバサイドにのみ存在し、参加者に直接配布しない
- 再識別結果は成功率（数値）のみ返却し、具体的なマッチング結果は非開示

### 9.5 倫理的配慮

- 「責任ある開示」モデルの採用: 再識別手法は対応する防御手法とセットでのみ公開
- 参加者教育: データプライバシーと再識別リスクに関する教育資料を提供
- 「匿名化の脆弱性発見」というフレーミングを用い、「攻撃成功」の過度な称賛を避ける

### 9.6 コミュニティ

- Discordサーバーまたは GitHub Discussions でのディスカッション場
- 大会終了後の上位チーム手法発表会（学会併設も検討）
- 過去問アーカイブの公開（翌年の新規参加者の練習用）

## 10. 設計上の重要判断ポイントまとめ

| 判断項目 | 決定 | 理由 |
|----------|------|------|
| データ | 合成データ | 法的リスク回避、評価の厳密性 |
| 提出形式 | Python関数（ファイルパスI/O） | 参加障壁の低さ、将来の多言語対応余地 |
| スコアリング | 有用性×安全性の積 | 片方を犠牲にする戦略の排除 |
| フェーズ | 匿名化→再識別の分離 | 公平性・計算量管理 |
| 初期実装 | ローカルモード優先 | 開発速度、サーバーは後から追加可 |
| 実行環境 | Docker サンドボックス | セキュリティ確保 |
| 再識別結果開示 | 成功率のみ | 攻撃手法精緻化リスクの低減 |
